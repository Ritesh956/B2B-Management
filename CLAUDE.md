# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

VendorHub is a B2B vendor management platform: PO approval workflows, invoice matching, contract tracking, and reporting across five roles (ADMIN, FINANCE, PROCUREMENT, MANAGER, VENDOR). Monorepo with two independent Node/TypeScript apps — `client/` (React) and `server/` (Express) — plus PostgreSQL and Redis.

Tech stack: React + TypeScript + Tailwind v4 + Zustand + TanStack Query (client); Express + Prisma + Socket.io + Bull/Redis + Zod (server); PostgreSQL 15, Redis 7; Docker Compose for containerized deploys.

## Deployment

- **Client**: Vercel, live at https://rit-vendor.vercel.app (project `client` under account `riteshgupta0510-6586`). Deployed via `npx vercel --prod` from `client/` (Vercel CLI was pre-authenticated in the deploying environment). `client/vercel.json` has a catch-all SPA rewrite (`/(.*) → /index.html`) — without it, direct navigation or a refresh on any non-root route (e.g. `/register`, `/vendor/pos/:id`) 404s, since Vercel serves the build as a static site by default and React Router never gets a chance to handle the route client-side.
- **Server**: Render, via the Blueprint at `render.yaml` (Postgres `vendorhub-db`, Redis `vendorhub-redis`, web service `vendorhub-server`). Live at https://vendorhub-server.onrender.com — auto-deploys on every push to `main`. Free tier: spins down after ~15 min idle, first request after that takes 30–60s to wake. `render.yaml` sets `PUBLIC_BASE_URL` (the API's own public origin — baked into uploaded-file URLs) and `CLIENT_URL` (the deployed client's origin, used for email links) directly as blueprint `envVars`; `SMTP_HOST`/`PORT`/`USER`/`PASS` are `sync: false` (must be filled in manually on the dashboard's Environment tab).
- `VITE_API_URL` is set as a Vercel env var on the `client` project, pointing at `https://vendorhub-server.onrender.com/api`, so the deployed client actually talks to the deployed server (not localhost).

### Deploy-blocking bugs found and fixed while first setting this up (2026-07-17)
These were pre-existing issues that only surfaced once a *real* production build/deploy was attempted — `npm run dev` uses `tsx`, which skips type-checking and module resolution differences entirely:
- `server/tsconfig.json` had `noEmit: true` — `npm run build` (`tsc`) exited 0 and reported nothing wrong, but never wrote a single file to `dist/`. Render's `npm start` (`node dist/index.js`) then failed with `Cannot find module`.
- Once emit was enabled, `module: "ESNext"` in the same tsconfig meant the emitted JS used `import`/`export` syntax while `package.json` declares `"type": "commonjs"` — Node refused to run it. Fixed by setting `module: "CommonJS"` (and dropping `moduleResolution`/`allowImportingTsExtensions`, which don't belong with CommonJS output).
- `server/src/index.ts`'s CORS origin whitelist only ever contained `localhost:5173`/`:3000` — every request from the deployed Vercel client was silently blocked by the browser's CORS check. Fixed then with a function-based origin check allowing any `*.vercel.app` subdomain plus the localhost origins; **tightened further on 2026-07-18** to a pinned allowlist (`server/src/config/cors.ts`: the real prod domain, this project's own Vercel preview URL pattern, `CLIENT_URL`, and localhost) since "any `*.vercel.app`" meant literally anyone's Vercel deployment could make credentialed cross-origin requests. Socket.io's CORS config uses the same allowlist now too.
- Missing `app.set('trust proxy', 1)` meant express-rate-limit keyed every request on Render's proxy IP instead of the real client IP (`ERR_ERL_UNEXPECTED_X_FORWARDED_FOR` in the logs) — all users shared one rate-limit bucket, so the strict 10-req/15min credential limiter 429'd real registrations almost immediately. Symptom in the UI: submitting the register form shows "Too many requests" and stays on the page.
- **Migration history had drifted badly from `schema.prisma`**: `prisma/migrations` only recorded a `Company` table and `User.companyId` (both long gone from the schema — removed at some point without ever running `prisma migrate dev` for it), and never recorded `isTwoFactorEnabled`, `status`, `lastLogin`, `isActive`, or the `deletedAt` soft-delete columns the app actually depends on today. Local dev only ever worked because the local DB was kept in sync via `prisma db push` directly against `schema.prisma`, which updates the DB but never touches migration history. `prisma migrate deploy` (what Render's build runs) only replays what's actually in the migrations folder, so the fresh Render Postgres ended up missing those columns — `register()` crashed with `The column User.isTwoFactorEnabled does not exist`. Fixed with a new migration (`20260717120000_sync_schema_drift`) generated by reconstructing the old migration history in a throwaway shadow database and diffing it against the current schema; verified the shadow DB matched `schema.prisma` exactly ("No difference detected") after applying it. **If you add new fields to `schema.prisma` going forward, use `prisma migrate dev` to generate a real migration — don't just edit the schema and rely on `db push` locally, or this same drift will happen again.**

The migration fix above was re-verified end-to-end (registration + login work in prod) in a follow-up session — deployment is healthy and stable as of 2026-07-17.

### PDF generation on Render — root cause and fix (2026-07-18)
`GET /pos/:id/pdf` and `GET /reports/export/monthly-pdf` (both use Puppeteer) 500'd in production with `Error: Could not find Chrome`. A first attempt added a `postinstall` script (`npx puppeteer browsers install chrome`) to `server/package.json`, but the build log showed the install *succeeding* every time while the runtime error persisted — a caching/skip theory turned out to be wrong. The actual cause: **Render builds and runs in separate containers**, and Puppeteer's default cache dir (`~/.cache/puppeteer`) lives outside the project directory, so it's simply not part of what gets shipped from the build container to the runtime one — only files inside the project directory (`dist/`, `node_modules/`, etc.) survive that handoff. Fixed with `server/.puppeteerrc.cjs` (Puppeteer's own documented fix for this exact CI/deploy scenario), pointing `cacheDirectory` at `server/.cache/puppeteer` — inside the project tree — so the install now lands somewhere that actually ships with the deploy. Verified live: both endpoints return `200` with valid PDFs. **If a Render deploy step "succeeds" but the runtime still fails the same way, check whether the artifact lives inside the project directory** — anything outside it doesn't survive the build→deploy split.

### Comprehensive audit remediation (2026-07-18)
A full pass through server + client fixing storage/money/auth/concurrency/scalability issues found in an end-to-end review — see commit `a3dcbb6` for the full list. The highlights that change how the app behaves (not just bug fixes) are documented inline under Architecture below: DB-backed OTP (`AuthToken` purpose `OTP_LOGIN`), current-password re-authentication on `/auth/me`, guarded (non-racy) PO approve/reject, one-active-invoice-per-PO + PO auto-close on payment, client-side route role gating, and `useSocket` finally being wired up. `server/src/config/s3.ts` was renamed to `storage.ts` — it never actually used S3, just local disk (`server/uploads/`), and the old name was actively misleading about durability (see "Uploaded files" below). Verified via `tsc` on both apps, full test suites, both production builds, a migration-history replay diffed clean against `schema.prisma`, and live end-to-end testing against the local server/DB.

A follow-up (2026-07-18, commit `4774f7e`) fixed a live symptom visible in Render's deploy history: repeated `Instance failed: HTTP health check failed with status code 429`. Render polls `/api/v1/health` frequently while waiting for a fresh instance to report healthy, and that polling counted against the same 200-req/15min general rate limiter as real traffic, occasionally exhausting it and getting an otherwise-healthy instance killed. Fixed by registering the health-check route before `app.use(generalLimiter)` in `index.ts`, so it's answered before ever reaching the limiter. **Any new route that Render or another piece of infrastructure polls directly (not a browser/API client) should go before the general limiter for the same reason** — don't just add it to the routes list further down.

(One existed briefly, holding only a `glob` devDependency for `migrate.js`, a one-off Tailwind dark-mode codemod — both were removed 2026-07-18 once their one job was done and its regex bug had been cleaned up; see the design system note under Architecture.)

## Architecture

### Auth & rate limiting
JWT-based auth. `server/src/middlewares/authenticate.ts` verifies the token, then rejects it outright if it carries `is2faPending` (the short-lived token `login()` issues before an OTP is verified — that flag existed for `verify-otp` to check, but nothing previously stopped it being used as a normal Bearer token everywhere else, a full 2FA bypass with just a password). It then looks up the user's **current** `role`/`isActive` from the DB on every request rather than trusting the token's embedded claims — tokens live up to 7 days with no revocation, so without this, deactivating a user or changing their role had no effect until their existing token expired. `findUnique` already excludes soft-deleted users via the Prisma extension (see below), so deletion is covered by the same check. This means `authenticate` costs a DB round-trip per request — consistent with the rest of the codebase's per-request lookups (e.g. the vendor-scoping pattern below), not a new pattern.

Two rate limiters exist in `server/src/index.ts`/`server/src/routes/auth.ts`: a generous general limiter (200 req/15min) applied globally, and a strict credential limiter (10 req/15min) applied *only* to `/auth/login`, `/auth/register`, and `/auth/verify-otp` — never to `/auth/me`, since that endpoint is polled on every page load by the client's `hydrate()` and would otherwise get rate-limited during normal use. Keep this split when touching auth routes. `GET /api/v1/health` (Render's `healthCheckPath`) is registered **before** `app.use(generalLimiter)` in `index.ts` for the same underlying reason — Render's own health-check polling was getting rate-limited and killing otherwise-healthy instances (see the 2026-07-18 note above). Any route hit by infrastructure rather than a real client belongs before the limiter, not after.

2FA login codes are persisted to `AuthToken` (`purpose: OTP_LOGIN`), not an in-memory `Map` — the old in-memory store didn't survive Render's free-tier sleep/restart between "login" and "enter the code arriving by email." The `token` column is namespaced per-user (`otp:<userId>:<code>`) so the raw 6-digit code alone is never a DB key on its own. A new login attempt invalidates any previous outstanding OTP for that user (`updateMany` on `usedAt: null` rows) before issuing a new one — always exactly one live code per user.

Changing your password or email via `PATCH /auth/me` requires re-submitting your **current** password (`currentPassword` in the body) — a stolen/leaked session token alone is no longer enough to take over the account by changing its login credentials. A wrong `currentPassword` returns **403, not 401** — this matters because the client's global axios response interceptor (`client/src/services/api.ts`) treats any 401 as "session invalid" and force-clears it; a 401 here would log the user out over a simple typo. If you add another endpoint that re-checks a credential the caller already authenticated with, use 403 for the "credential didn't match" case, not 401.

Client-side, `client/src/store/authStore.ts`'s `hydrate()` only clears the session on a genuine 401/403 from `/auth/me` — not on 429s or network errors, which are treated as transient. The store's `isLoading` starts `true` whenever a token exists in `localStorage`, and `App.tsx` blocks rendering the router until it resolves; this exists specifically so role-gated route guards (`ProtectedRoute`, `VendorOnlyRoute`) never evaluate `user` before `hydrate()` has had a chance to populate it. **`isLoading` belongs to `hydrate()` alone** — `login()`/`verifyOtp()` must never set it, because App.tsx unmounts the entire router while it's true, which killed the login page mid-submit and made its `navigate()` a no-op (every first login bounced back to `/login` until this was fixed). A genuine 401 from *any* API call (not just `/auth/me`) is handled the same way: `api.ts`'s response interceptor clears the token and calls a handler `authStore` registers at startup (`registerUnauthorizedHandler`) rather than `window.location.href`-ing — a full page reload used to blow away the entire SPA over one expired token; now `ProtectedRoute` just redirects client-side on the next render.

### Password reset & user invites are DB-backed, not in-memory
Both flows mint a random token, persist it to the `AuthToken` model (`purpose: PASSWORD_RESET | INVITE`, `expiresAt`, `usedAt`), and email a link built from `CLIENT_URL` (falls back to the deployed client's real domain if unset — never hardcode `localhost` here, that was the original bug). Do **not** revert to an in-memory `Map` for either of these — Render's free tier restarts/sleeps wipe process memory, so a Map-based token is dead by the time a real user clicks an emailed link. `resetPassword`/`acceptInvite` mark the token `usedAt` inside a `$transaction` alongside the user update, enforcing single-use; `resetPassword` additionally invalidates any other outstanding `PASSWORD_RESET` tokens for that user, so an older reset email in someone else's hands stops working the moment a newer one is used. The in-memory OTP store this section used to warn about is gone — 2FA codes are DB-backed too now (see "Auth & rate limiting" above).

Inviting a user to a staff role (`POST /users/invite`) **rejects `role: VENDOR`** — an invited vendor has no company name/phone collected by the invite form, so no `Vendor` row gets created, and their portal 404s everywhere ("Vendor profile not found") the same way pre-2026-07-17 self-registered vendors did. Vendors must self-register instead (`POST /auth/register`), which creates both rows together. Re-inviting an email stuck in `UserStatus.INVITED` (e.g. they never accepted, or the link expired) re-issues a fresh invite token rather than 409-ing — the old behavior dead-ended an expired invite permanently with no recovery path other than deleting and recreating the user.

Both `updateUserRole` and `deactivateUser`/`deleteItem` (admin.ts) refuse to let the last active `ADMIN` demote, deactivate, or delete themselves, and refuse to let anyone demote/deactivate/delete the *only* remaining active admin — those actions would permanently lock the org out of `/admin/users` and every other admin-only surface with no recovery path except direct DB access. If you add a new admin-destructive action, apply the same "count other active admins first" guard.

### Uploaded files are served through an authenticated route, not express.static
`server/src/controllers/files.ts` (mounted at `/uploads` in `index.ts`) replaced a bare `express.static` that had zero authentication — anyone with (or guessing) a file URL could read any invoice/contract/vendor-document PDF. Vendor-onboarding documents are staff-only (same rule as the vendor directory below); invoice/contract PDFs check the requester either has staff access or owns the vendor record the file belongs to. Since `<a href>`/`<iframe src>` can't send an `Authorization` header, the route also accepts the session token as a `?token=` query param — `client/src/utils/fileUrl.ts`'s `withAuthToken()` appends it wherever one of these URLs is rendered. Follow the same pattern (route auth + `withAuthToken`) for any new file-serving endpoint; don't add another `express.static` mount.

`server/src/config/storage.ts` (renamed from `s3.ts` on 2026-07-18 — it never actually used S3) is local-disk multer storage, not a real object store. `PUBLIC_BASE_URL` gets baked into every uploaded file's `fileUrl` so the URL is at least reachable from the deployed client, but the files themselves live on Render's local disk, which is **ephemeral on the free tier** — every deploy/restart wipes `server/uploads/`. Durable storage needs a real object store (S3/R2, using `storage.ts`'s existing multer plumbing as the integration point) or a Render persistent disk; neither is wired up yet. `server/uploads/` is gitignored (previously committed PDFs were untracked on 2026-07-18) — don't re-add it to git, and don't treat anything in it as durable in production.

### Client styling is one system: CSS custom properties, not Tailwind utility classes
Despite Tailwind v4 being installed (`client/src/index.css` still has `@import "tailwindcss"` and the `@custom-variant dark` wiring), the client's actual design system is that same file's CSS custom properties (`--bg-card`, `--text-primary`, `--accent-primary`, `--border-dim`, etc., themed via a `.light`/`.dark` class the `ThemeProvider` toggles on `<html>`) plus shared classes — `.card`, `.btn-primary`/`.btn-secondary`/`.btn-ghost`/`.btn-danger`, `.input-base`, `.modal-backdrop`/`.modal-box`, `.badge-*`, `.data-table`, `.page-root`/`.page-title`/`.page-subtitle`, `.stat-card`. Most components apply these via inline `style={{}}` objects rather than `className` utility strings.

This wasn't always consistent — 11 files were built with raw Tailwind utility classes instead of the token system, and a one-off codemod (`migrate.js`, since removed) later bolted dark-mode `dark:` variants onto just those files via non-idempotent string regex. Re-running (or double-matching) produced duplicated `dark:X dark:Y` class pairs, which caused at least one live production bug: `InviteUserModal`'s backdrop rendered solid opaque white instead of a translucent dark blur, because the stray `dark:bg-white` won the cascade over the intended `dark:bg-slate-950/60`. All 11 files (`ActivityFeed`, `BulkActionBar`, `CopyToClipboard`, `DeletedItemsSection`, `GlobalErrorBoundary`, `KeyboardShortcutsModal`, `ProtectedRoute`, `AcceptInvitePage`, `InviteUserModal`, `VendorInvoiceNewPage`, `VendorPerformancePage`) were migrated onto the CSS-variable system on 2026-07-18. **When adding a new component, follow the CSS-variable/shared-class pattern — don't reach for Tailwind utility classes**, even though the Tailwind dependency itself is still installed (removing it entirely — the PostCSS/Vite plugin wiring — wasn't in scope for this cleanup, since nothing currently depends on it).

### Client bundle & CSP
Every route mounted under `ProtectedRoute` in `client/src/App.tsx` is `React.lazy()`-loaded behind a single top-level `<Suspense>` — only the public auth-flow pages (`LoginPage`, `RegisterPage`, `VerifyOtpPage`, `ForgotPasswordPage`, `ResetPasswordPage`) stay eagerly bundled, since they're what an unauthenticated visitor loads first anyway. This was added 2026-07-18 specifically because Recharts (pulled in by `DashboardPage`/`ReportsPage`/`VendorPerformancePage`) was the single biggest contributor to a 1.2MB single-chunk bundle; it's now its own ~350KB chunk that only loads when a chart page is actually visited (main chunk dropped to ~468KB). **When adding a new page component to `App.tsx`, add it via `lazy(() => import(...))` like the others** — don't import it eagerly at the top, or it defeats the split.

`client/vercel.json` sets security headers including a CSP. `style-src` includes `'unsafe-inline'` deliberately — the app's entire design system relies on inline `style={{}}` objects (see "Client styling" below), and there's no SSR to inject per-request nonces for a strict `style-src`. `script-src` is `'self'` only (no inline scripts, no external script tags) and `connect-src` is pinned to the known API/Sentry origins — if you add a new external service the client talks to directly (analytics, a different API host, etc.), it needs adding to `connect-src` or the browser will silently block it.

### Client-side route role gating (App.tsx)
`ProtectedRoute` only checks "is there a token" — it has no idea about roles. Every route mounted under `AppLayout` in `client/src/App.tsx` is additionally wrapped in a `RoleRoute` component that mirrors the backend's actual `authorize()` list for that resource (e.g. `/invoices` → `[ADMIN, FINANCE]`, `/reports` → `[FINANCE, ADMIN]`, `/audit-logs` → `[ADMIN]`). Before 2026-07-18, a VENDOR (or a wrong-role staff member) hitting one of these routes directly by URL would load the staff shell and get an empty page backed by a wall of 403s instead of being redirected. `RoleRoute` sends a mismatched VENDOR to `/vendor/dashboard` and everyone else to `/dashboard`. **When adding a new staff-only route, wrap it in `RoleRoute` with the same role list the backend route actually enforces** — don't rely on the sidebar nav's role filter alone, since that only hides the link, it doesn't stop direct navigation.

### Two parallel app shells
Staff roles (ADMIN/FINANCE/PROCUREMENT/MANAGER) and VENDOR use **separate layout components and separate route trees** in `client/src/App.tsx`: `AppLayout` (staff sidebar/topbar) vs `VendorLayout` (vendor portal chrome), gated by `ProtectedRoute` + `VendorOnlyRoute` respectively. Some pages are deliberately duplicated across both trees (e.g. `PODetail` is mounted at both `/pos/:id` and `/vendor/pos/:id`) rather than shared via a single route, because the two shells need different back-navigation targets and nav visibility. When adding a page a vendor should be able to reach, mount it under the `/vendor` route tree with `VendorLayout` — don't just link a vendor to a staff route.

Each layout wraps only its own `<Outlet />` in an `<ErrorBoundary key={location.pathname}>` (not the whole layout) — a crash in one page must not take down the sidebar, and keying on pathname makes the boundary self-heal on navigation instead of showing a stale error after the user clicks away.

### Role scoping pattern (Vendor ↔ User is by email, not FK)
There is no foreign key between a `User` (role=VENDOR) and its `Vendor` record — they're linked by matching `email`. Every backend controller that needs to scope data to "the vendor currently logged in" follows the same pattern: look up the `User` by `req.user.id` to get their email, then filter `where: { vendor: { email } }` (see `listPOs`, `listInvoices`, `listContracts` in `server/src/controllers/`). When adding a new vendor-facing list/detail endpoint, follow this exact pattern — a missed scope here means one vendor can see another vendor's data. `register()` creates both rows together in one transaction for a self-registered VENDOR — the `Vendor` row didn't exist before 2026-07-17, so every self-registered vendor's own portal 404'd ("Vendor profile not found") until that email-based link had something to actually match.

**The vendor directory itself (`GET/PATCH /api/v1/vendors*`) and the company-wide dashboard (`GET /api/v1/dashboard/*`) are staff-only** (`ADMIN`/`PROCUREMENT`/`MANAGER`/`FINANCE`) — a `VENDOR` has no legitimate reason to browse other vendors or see cross-tenant activity/spend data, and until 2026-07-17 neither router had a role check at all (only `authenticate`), so any vendor could read every other vendor's contact info and the full company activity feed. Vendors manage their own record through `/api/v1/vendor/profile` and see their own activity through `/api/v1/vendor/dashboard` instead. If you add a new route to either router, it inherits the router-level `authorize(STAFF_ROLES)` by default — don't bypass it for a vendor-facing need without scoping the query first.

### PO approval chains
Approval chains are dynamic and amount-tiered (`server/src/services/approvalService.ts`, `getApprovalRoles`): POs over ₹500k require MANAGER → FINANCE → ADMIN sign-off, over ₹50k require MANAGER → FINANCE, otherwise just MANAGER. The chain is stored as a JSON blob on the `PurchaseOrder` row (`approvalChain` + `currentApproverIndex`), not as separate relational rows — `approveState`/`rejectState` in the same file are the only functions that should mutate it.

`approvePO`/`rejectPO` (`server/src/controllers/pos.ts`) write via a **guarded `updateMany`** — `WHERE id = ? AND status = 'PENDING_APPROVAL' AND currentApproverIndex = <the value just read>` — and return `409` if it affects zero rows, instead of an unconditional `update`. The old read-then-write let two approvers (or one approver double-clicking/double-submitting) both read the same `currentApproverIndex` and both write, silently double-advancing the chain. Any future mutation of `approvalChain`/`currentApproverIndex`/`status` must go through the same guarded-write pattern — don't simplify it back to a plain `findUnique` + `update`.

### Invoice/PO money-integrity rules
`submitInvoice` (`server/src/controllers/invoices.ts`) enforces **one active invoice per PO** — a PO with an existing `SUBMITTED`/`MATCHED`/`APPROVED`/`PAID` invoice rejects a new submission with `409`. Before this, a vendor could submit unlimited invoices against a single approved PO with no spend ceiling at all; a prior `MISMATCHED` invoice doesn't block resubmission (there's no invoice-rejection flow, so that's the vendor correcting their own mistake). The vendor's own invoice number (their accounting reference) is honored when provided instead of always auto-generating one — it used to be collected by the form and silently discarded.

`payInvoice` transitions the PO to `POStatus.CLOSED` in the same `$transaction` as marking the invoice `PAID` — `CLOSED` was a defined-but-unreachable enum value before 2026-07-18; a fully-paid PO stayed `APPROVED` forever, which (combined with the one-invoice-per-PO rule above) also meant it kept silently accepting new invoice submissions after payment. If you add a new PO/invoice status transition, check whether it should also flip the other side of the relationship the same way.

### Vendor performance score scale
`Vendor.performanceScore` is 0–5 everywhere in the product (displayed as e.g. "4.6" or "4.6 / 5") — never 0–100. Both the backend validator (`server/src/controllers/vendors.ts`, `updateVendorPerformanceScore`) and the client's color/bar-width math (`client/src/pages/Vendors/VendorPerformancePage.tsx`) must stay on this scale.

### Soft delete is a Prisma extension, not a convention you enforce per-query
`server/src/config/prisma.ts` extends the client so `findMany`/`findFirst`/`findUnique`/`count`/`groupBy`/`aggregate` on `User`/`Vendor`/`PurchaseOrder`/`Invoice`/`Contract` automatically exclude `deletedAt IS NOT NULL` rows — you don't (and shouldn't) add that filter yourself in a controller. If you add a new query method usage Prisma supports that isn't already wrapped there (the extension only covers the six listed above), soft-deleted rows will leak back in silently; extend the list rather than filtering ad hoc in the controller.

This is also why `generatePONumber`/`generateInvoiceNumber` (`server/src/controllers/pos.ts` / `invoices.ts`) use a raw `$queryRaw` instead of a normal Prisma query: a soft-deleted PO/invoice still holds its number in the unique index, but the extension would hide it from a normal `findFirst`, so "next number" kept getting recomputed as an already-used one and collided. Both also wrap their `create` call in `retryOnUniqueConflict` (`server/src/utils/retryOnUniqueConflict.ts`) as a second layer, since the read-then-write is still a race under real concurrency even with the raw query. Don't simplify either of these back to a plain `findFirst` + `create`.

### Background jobs & realtime
Bull queues (`server/src/queues/`) handle email and notification delivery; `redisAvailability.ts` probes Redis with a raw TCP connect (its own 300ms timeout) and lets queue initialization degrade gracefully if Redis is unreachable, rather than crashing the server. That probe's result is cached with a TTL (60s on success, 10s on failure) rather than forever — the original version cached the very first check permanently, so a Redis that wasn't reachable yet at boot (a routine cold-start race) disabled email/notifications for the rest of the process's life. `GET /api/v1/health` uses the same probe (see "Auth & rate limiting" above) instead of pinging through Bull's ioredis client directly — ioredis's default retry strategy means a direct `.ping()` never rejects while Redis is down, it just queues and waits, which is what made the health endpoint hang indefinitely before 2026-07-18.

Socket.io (`server/src/utils/socket.ts`) authenticates each connection via the same JWT (and, since 2026-07-18, applies the same `is2faPending`/deactivated-user checks as the HTTP `authenticate` middleware, plus the same CORS origin allowlist from `config/cors.ts` instead of `origin: '*'`) and joins a room named after the user's id. The client side (`client/src/hooks/useSocket.ts`) is now actually called — once each, from `AppLayout` and `VendorLayout` — and pushes incoming `notification` events into `client/src/store/notificationStore.ts`, a small Zustand store shared with `NotificationBell`'s polling fallback (60s interval, just a catch-up in case the socket connection dropped) so both write into the same list instead of drifting out of sync. `notifyUser` (`server/src/utils/notify.ts`) optionally takes a `{ entity, entityId }` link (matching the `Notification.entity`/`entityId` columns) — `NotificationBell` resolves that into a role-aware route (staff vs `/vendor/*`) and renders a "View →" click-through. Not every `notifyUser` call passes a link (e.g. the contract-expiry sweep notifies about a batch of contracts, not one record) — that's expected, the bell just doesn't render a link for those.

### Audit log access is role/ownership-scoped, not a flat ADMIN gate
`GET /api/v1/audit-logs` (`server/src/controllers/auditLogs.ts`) serves two different callers with different rules: the standalone Audit Logs page (ADMIN-only, unscoped browsing with optional filters) and the embedded `ActivityFeed` widget (`client/src/components/ActivityFeed.tsx`, used on PO/Invoice/Vendor detail pages across both app shells, always passes a specific `entity`+`entityId`). Non-ADMIN callers are only allowed through when both `entity` (one of `PurchaseOrder`/`Invoice`/`Vendor`) and `entityId` are present; VENDOR additionally has to own that specific record (checked by resolving the record's vendor email and comparing to the caller's). Staff scoping (added 2026-07-18) mirrors what each role can actually open elsewhere rather than assuming "staff = unrestricted": `Invoice` history is FINANCE-only (matches invoice detail pages being FINANCE/ADMIN-only — MANAGER/PROCUREMENT get a 403), and PROCUREMENT's `PurchaseOrder` history is limited to POs they created themselves (matches `getPOById`'s own scoping). If you add a new entity type to `ActivityFeed`, add it to `SCOPED_ENTITIES` and teach `getOwningVendorEmail` how to resolve it, and check whether any staff role needs the same "only what they can already view directly" restriction — don't assume all staff roles get unrestricted access by default.
